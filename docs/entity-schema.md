## Common Fields

Every entity has these computed fields. They are system-managed and not set by application code.

| Field | Type | Rules |
|-------|------|-------|
| created_at | timestamp | set on insert; never modified |
| updated_at | timestamp | set on insert; auto-updated on every modification |

These fields are omitted from individual entity tables below for brevity.

---

## User

### Fields
| Field | Type | Rules |
|-------|------|-------|
| id | UUID | system-generated |
| role | enum | `client`, `agent`, `admin` |
| email | string | unique; required |
| display_name | string | optional; can be set later via profile update |
| avatar_url | string | optional |
| locale | string | default `en-AU` |
| is_active | boolean | default true; soft deactivation for all roles |
| google_id | string | optional; Google account subject ID for SSO |

### Business Rules
- `email` is required for all users
- `google_id` is set when a user signs in via Google; links the Google account to the user
- A user cannot change their own role
- Deactivation is soft — set `is_active=false`, never hard delete

## OtpCode

### Fields
| Field | Type | Rules |
|-------|------|-------|
| id | UUID | system-generated |
| email | string | email address |
| code | string | 6-digit numeric |
| expires_at | timestamp | 5 minutes from creation |
| verified_at | timestamp | null until successfully verified |

### Business Rules
- Only one active (unverified, unexpired) OTP per email at a time
- Creating a new OTP invalidates any previous one for the same email
- A verified OTP cannot be used again
- OTP codes older than 24 hours should be purged (background job)

## Chat

### Fields
| Field | Type | Rules |
|-------|------|-------|
| id | UUID | system-generated |
| title | string | optional; auto-generated from first message if not set |

### Business Rules
- A chat must always have at least one `client` participant

## ChatParticipant

### Fields
| Field | Type | Rules |
|-------|------|-------|
| chat_id | UUID | references Chat |
| user_id | UUID | references User |
| role | enum | `client`, `agent`, `observer` |
| joined_at | timestamp | when user joined the chat |
| last_read_at | timestamp | for unread count calculation |

### Business Rules
- A user can only appear once per chat
- The creating client is automatically added as `client` role on chat creation
- Human agents join with role `agent`; admins monitoring join as `observer`
- Observers can read but not send messages
- A chat must always have at least one `client` participant

## Message

### Fields
| Field | Type | Rules |
|-------|------|-------|
| id | UUID | system-generated |
| chat_id | UUID | references Chat |
| sender_id | UUID | references User; null for system messages |
| sender_role | enum | `client`, `agent`, `system` |
| content_type | enum | `text`, `image`, `attachment`, `video`, `tool` |
| content | JSONB | shape depends on content_type (see below) |
| label | string[] | optional; values: `order`, `action_required`, `payment`, `shipment`, `delivery` |

### Content shapes by type
```
text:         { text: string }
image:        { url: string, thumbnail_url?: string, caption?: string }
attachment:   { url: string, filename: string, size_bytes: number, mime_type: string }
video:        { url: string, thumbnail_url?: string, duration_seconds?: number }
tool:         { tool: string, data: object }
```

### Business Rules
- Messages cannot be edited or deleted
- `system` role messages are generated by the server (e.g. "Agent joined the chat")
